[연구 일지] 2026-03-11

1. 주요 내용
- 멀티 리전 재해 복구(DR) 아키텍처 확장 및 웜 스탠바이(Warm Standby) 기반 자동 트래픽 우회(Failover) 검증.

2. 진행 요약
- 서울 리전(Main) 장애 발생 시 도쿄 리전(Standby)으로 트래픽을 자동 우회하도록 테라폼 코드를 확장함.
- 도쿄 리전에 최소 인스턴스 1대를 유지하는 웜 스탠바이 전략을 적용하여, 글로벌 로드밸런서가 다운타임 없이 트래픽을 넘길 수 있도록 인프라를 구성함.
- Locust를 활용해 100명 규모의 소규모 트래픽을 발생시킨 상태에서 서울 리전의 오토스케일러 가용량을 0으로 강제 축소하여 장애 상황을 시뮬레이션함.
- 장애 발생 시 500 에러 없이 도쿄 리전으로 접속이 전환(Failover)되는 것을 확인하고, 서울 리전 복구 후 다시 기존 서버로 트래픽이 원복(Failback)되는 전체 라이프사이클을 검증함.
- 실제 운영 환경의 DR 흐름과 비교 분석하여, 무상태(Stateless) 웹 서버 계층의 복구 외에도 상태 저장(Stateful) 데이터베이스 계층의 동기화 및 병합 작업이 실무에서 추가적으로 요구됨을 확인함.

3. 주요 수정 및 개발 내용
[네트워크 인프라 확장] 
- modules/network: 도쿄 리전(asia-northeast1) 전용 서비스 서브넷(10.0.30.0/24) 및 Cloud NAT 리소스 추가.

[컴퓨팅 및 로드밸런서 인프라 확장] 
- modules/compute: 도쿄 리전 환경에 맞춘 인스턴스 템플릿, 관리형 인스턴스 그룹(MIG), 오토스케일러 리소스 추가.
- modules/lb: 글로벌 로드밸런서의 백엔드 서비스 그룹에 도쿄 리전 MIG를 연동하여 멀티 리전 트래픽 분산 구성.

[테라폼 모듈 구조 개선]
- root main.tf / variables.tf: 모듈 간 출력값(Output) 참조 구조를 개선하여, 네트워크 모듈에서 생성된 도쿄 서브넷 ID가 컴퓨팅 모듈로, 컴퓨팅 모듈의 도쿄 MIG 정보가 로드밸런서 모듈로 정확히 전달되도록 변수 체계 리팩토링.

4. 시행 착오 및 오류 해결
[테라폼 루트 변수 주입 오류]
- 현상: 루트 variables.tf 파일에 내부 모듈 간 전달되어야 할 출력값(network_name, subnet_serving_id 등)이 외부 주입 변수로 잘못 선언되어 구조적 오류 발생.
- 해결: 루트 파일에는 외부 주입용 핵심 변수(project_id, region 등)만 남기고, 모듈 간 파라미터 전달은 module.network.subnet_serving_id 형태로 직접 참조하도록 연결 구조를 수정함.

[Locust 부하 테스트 시 403 Forbidden 에러 발생]
- 현상: 트래픽 주입 시 서버 정상 응답 대신 Locust 차트에 403 에러가 폭증함 (반면 로컬 PC 웹 브라우저 접속은 정상 작동).
- 분석 및 해결: 기존에 설정해 둔 Cloud Armor의 Rate Limiting 규칙(단일 IP 1분당 100회 초과 시 차단)이 정확히 작동한 결과임을 확인함. Cloud Shell 가상 머신의 단일 IP로 다량의 요청이 발생하여 차단된 것이며, 테스트 진행을 위해 임시로 해당 방어 규칙의 허용 한도를 10,000회로 상향 적용하여 문제를 해결함.

[CLI 명령어 구문 및 환경 호환성 오류]
- 현상: 인스턴스 그룹 축소 명령어(autoscalers update) 오작동 및 윈도우 CMD 환경에서 로드밸런서 로그 조회 시 따옴표 파싱 에러(INVALID_ARGUMENT: Field not found) 발생.
- 해결: 축소 명령어를 인스턴스 그룹 직접 타겟팅(instance-groups managed update-autoscaling) 방식으로 수정하고, 윈도우 환경에 맞춰 작은따옴표를 큰따옴표로 치환하여 정상 실행함.

[Failover 시 500 에러 미발생 원인 분석]
- 현상: 서울 리전 인스턴스 강제 축소 시 접속 유실(500 에러)이 관측되지 않고 매끄럽게 도쿄 리전으로 전환됨.
- 분석: 도쿄 리전에 서버가 선제적으로 대기 중(Warm Standby)이었으며, 서울 리전 인스턴스가 즉시 소멸되지 않고 기존 연결을 안전하게 종료하는 '연결 드레이닝(Connection Draining)' 상태를 유지하며 헬스체크에 응답했기 때문임을 확인함.
